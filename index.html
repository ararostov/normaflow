Создание заказа на сайте / в приложении

Описание

Три блока в одном UI-сценарии: корзина с реалтайм-скидками из Maxma, ввод адреса забора с получением зоны и слотов из Телетрека (адрес отдачи опциональный), финальное создание заказа параллельно во всех системах.

Фазы

  • Блок AКорзина и реалтайм-скидки. Клиент собирает вещи, Back запрашивает скидки у Maxma на каждое изменение корзины.
  • Блок BАдреса и слоты. Клиент вводит адрес забора (обязательный); Back идёт в Телетрек за зоной, считает слоты. Адрес отдачи опциональный.
  • Блок CСоздание во всех системах. Клиент нажимает «Оформить»; Back параллельно регистрирует заказ в АГБИС, задачу в Телетреке, заказ в Maxma.

Описание

Альтернатива Этапу 1. Ведущая система — АГБИС. Технолог оформляет заказ прямо в ПП (клиент с вещами физически здесь). Адрес доставки не заполняется — клиент выберет в приложении. Подтверждение «фабрика = Да / клиент = Да» ставится сразу. Лояльность (скидки, бонусы, промокоды) применяется в АГБИС (локальная копия), затем Back синхронизирует с Maxma. Оплата возможна в ПП сразу — либо отложить до приложения или при получении.

Фазы

  • Фаза 1Технолог оформляет заказ в АГБИС. Состав, цена, опционально фото. Адрес не заполняется.
  • Фаза 2Лояльность и оплата в ПП. ALT оплатил в ПП (наличные / карта / СБП АГБИС) · ELSE отложил оплату.
  • Фаза 3Подтверждение сразу. Технолог ставит фабрика = Да и клиент = Да одновременно.
  • Фаза 4Синхронизация в Back с задержкой 15–45 мин. Заказ приходит в Back со всеми признаками. Сразу в BACK-5. Back катит в Maxma: скидку и списание бонусов.

Влияние на другие этапы

Этап 2 пропускается (вещи уже в ПП). Этап 3 пропускается (уже подтверждено). Этап 4 частично (если оплатил — пропускается). Этапы 5, 7 идентичны. Этап 6 — выбирается 6А или 6В в приложении (или устно в ПП).

На выходе
AG-1 · BACK-5 · MAX-1 · фабрика = Да · клиент = Да

Описание

Ведущая система — Телетрек. Он сам принимает задачу, назначает курьера и проводит его по статусам. Back только слушает события и реагирует: обновляет свой статус и триггерит виджеты / уведомления в приложении клиента.

Фазы

  • Фаза 1Телетрек принимает задачу. TELE-1 → TELE-2. Для Back тихо.
  • Фаза 2Назначен курьер. Back: BACK-2 «Назначен курьер (приём)», виджет W-1 включается, App: APP-1 → APP-3 «Сбор вещей».
  • Фаза 3Курьер выехал. TELE-3. Back: BACK-3 «Сбор вещей (приём)», push N-1. Статус App не меняется.
  • Фаза 4Курьер в пути к клиенту. TELE-4. Мы игнорируем — у Back и App ничего не меняется.
  • Фаза 5Курьер забрал вещи. TELE-5. Виджет W-1 выключается. Статус App остаётся APP-3 (дальше меняет АГБИС на Этапе 3).

Описание

АГБИС — ведущая система, но с задержкой 15–45 минут на отправку данных в Back (отправка в АГБИС быстрая). Технолог фотографирует вещи, добавляет комментарии и доп. реквизиты, корректирует цену, подтверждает. Клиент видит обновлённый заказ и тоже подтверждает.

Фазы

  • Фаза 1Технолог готовит заказ в АГБИС. Фото, комментарии, реквизиты, корректировка цены. Данные с задержкой долетают до Back → App.
  • Фаза 2Технолог подтверждает. Признак «фабрика = Да» в АГБИС. С задержкой в Back: BACK-4, App: APP-4, виджет W-2 «Подтвердите заказ» включён.
  • Фаза 3Клиент подтверждает в приложении. App: APP-5, Back: BACK-5, признак «клиент = Да» быстро уходит в АГБИС.

Статусы заказа не меняются. Меняются только признаки оплаты. Оплата собирается из скидки, промокода, бонусов и денежного остатка. Остаток — один из трёх сценариев. Если сценарий C (при доставке) — готовим QR Cloud для курьера, Life готовить не надо (работает через матчинг).

02 · Акторы

Два типа пользователей, взаимодействующих с флоу.

Клиент

Конечный пользователь химчистки
  • Взаимодействует с флоу только через Norma App
  • Оформляет заказ, выбирает слоты, подтверждает приёмку, оплачивает, оставляет отзыв
  • Не видит: статусы АГБИС (только детализацию услуг), фотоотчёты курьера, внутренние признаки

Технолог

Сотрудник фабрики
  • Работает только через АГБИС
  • Принимает заказ (фото, комментарии, коррекция цены), подтверждает
  • Ведёт услуги через статусы (в работу / завершить / отменить)
  • Может сгенерировать ссылку на оплату для клиента (Сценарий B, Этап 4)

03 · Системы

Семь технических систем, задействованных во флоу. Нажмите на название — развернётся описание.

Norma AppКлиентский UI
  • Единственная точка взаимодействия клиента с флоу
  • Статусы APP-1…APP-10, виджеты W-1, W-2, уведомления N-1
  • Оформление, выбор адреса / слота, конфигурация оплаты, отзыв
  • Все данные — только через Norma Back
Norma BackОркестратор · источник правды по заказу
  • Центральная точка всего флоу. Все остальные системы общаются через Back
  • Основной статус заказа BACK-1…BACK-11, признаки оплаты и подтверждения
  • Синхронизирует статусы между всеми системами
  • Содержит модуль матчинга Life: сопоставляет платежи с заказами по сумме / времени / курьеру
  • Хранит фотоотчёты курьера и отзывы клиентов
АГБИСФабричная учётная система
  • Технолог ведёт заказ через АГБИС
  • Отслеживает статусы услуг (AG-I-1…AG-I-6) отдельно по каждой вещи
  • Автоматически выводит статус заказа (AG-1…AG-6) из статусов услуг
  • Хранит признаки подтверждения (фабрика / клиент) и оплаты
  • задержка 15–45 мин при отправке данных в Back. Отправка в АГБИС — быстрая
ТелетрекКурьерская система
  • Управляет курьерами и задачами доставки (забор и отдача)
  • Определяет зону по адресу и выдаёт доступные слоты
  • Ведёт задачу через TELE-1…TELE-5, сама назначает исполнителя
  • Хранит QR-коды для оплаты СБП курьеру
  • Передаёт фотоотчёты и подписи клиентов
  • Реалтайм-интеграция с Back
MaxmaCRM и программа лояльности
  • Баланс бонусов клиента, история скидок, промокоды
  • Рассчитывает скидки по количеству / сумме в корзине (реалтайм)
  • Списывает бонусы, применяет скидки и промокоды одним вызовом
  • Статусы заказа MAX-1…MAX-3 для собственного учёта
  • Отзывы клиентов не передаются (хранятся только в Back)
Cloud · CloudPaymentsОнлайн-оплата
  • Генерирует ссылки на оплату (карта / СБП) по запросу Back
  • Проводит платёж, присылает колбэк «заказ оплачен»
  • Используется в трёх местах: онлайн сейчас, ссылка от технолога, QR курьеру
Life · LifePayТерминальная оплата картой
  • Принимает оплату картой через терминал курьера
  • Присылает в Back уведомление о платеже (сумма, время, терминал / курьер)
  • Не знает о заказе — Back сам сопоставляет через модуль матчинга
  • Используется только в сценарии C-КАРТА при доставке

Критическое · UX + надёжность

Задержка синхронизации с АГБИС 15–45 минут

АГБИС возвращает данные в Back с задержкой 15–45 минут (внутренний цикл АГБИС). Это влияет на Этапы 3 и 5:

  • Клиент может ждать до 45 минут появления факта «технолог принял заказ»
  • Виджет W-2 "Подтвердите заказ" появляется с большим лагом
  • Переход APP-6 → APP-7 тоже с задержкой
Что делать: продуктово — пояснять клиенту в UI («данные обновляются в течение часа»). Технически — мониторинг задержек, SLA на стороне АГБИС, промежуточный кэш или polling чаще. Долгосрочно — договариваться с АГБИС об ускорении выгрузки или webhooks.
Критическое · Консистентность данных

Два источника правды по бонусам — Maxma vs АГБИС

Maxma — глобальный мастер программы лояльности. АГБИС ведёт локальную копию балансов для работы в ПП (Этап 1Б) и оплат на месте. Между системами происходит синхронизация через Back, но она асинхронная и с задержкой 15–45 мин (цикл АГБИС). Это реальный риск, не гипотетический — в сценарии с оплатой в ПП балансы фактически списываются из АГБИС и только потом катятся в Maxma.

Возможные проблемы:

  • Двойное списание бонусов: клиент оплатил бонусами в ПП и одновременно (или сразу после) в приложении, пока синхронизация не дошла. Два раза списали одни и те же бонусы.
  • Рассинхрон балансов: в Maxma баланс 500, в АГБИС 450 (последнее списание ещё не синхронизировано). Клиент в приложении думает что может списать 500, но АГБИС говорит «только 450».
  • Промокод использован дважды: одноразовый промокод применён в АГБИС в ПП и в Maxma через приложение до синхронизации.
  • Акции по количеству заказов: рассинхрон счётчиков может дать клиенту лишнюю акцию или отобрать заслуженную.
Что делать: мониторинг рассинхронов (метрика «расхождение баланса Maxma vs АГБИС»), ручная политика разрешения конфликтов (Maxma — победитель, АГБИС подстраивается), блокировки одновременного использования бонусов (lock в Back на время открытой транзакции). Для промокодов — сразу после применения в одной системе помечать как использованный в обеих через Back. Нельзя исключить полностью при текущей архитектуре АГБИС — нужно жить с этим и мониторить.
Высокое · Консистентность данных

Два источника правды по скидкам

Аналогично бонусам: Maxma рассчитывает скидки по количеству / сумме, но в АГБИС есть свои скидки, которые технолог может применить вручную. Если он это сделает, в Back и Maxma не узнают.

Что делать: запретить применение скидок АГБИС напрямую в текущем флоу. Все скидки — через Maxma на Этапе 4. Или наладить обратную синхронизацию АГБИС → Back → Maxma.
Критическое · Качество продукта

Слабая проработка отмен и unhappy paths

Текущий флоу описывает только happy path. Отсутствуют:

  • Клиент не подтверждает на Этапе 3 → возврат вещей
  • Одна или несколько услуг отменены на Этапе 5 → возврат разницы
  • Cloud-платёж упал / клиент бросил оплату → повторная попытка или откат
  • Промокод устарел между проверкой и применением
  • Клиент отказывается на доставке → возврат на фабрику + возврат денег
  • Курьер не смог забрать или вручить
  • Терминал Life не работает / СБП не проходит
Что делать: отдельный документ «Unhappy paths» с матрицей: что где может сломаться, какие статусы (BACK-12…BACK-16) использовать, как возвращать деньги, как восстанавливать состояние.
Высокое · Финансы

Возврат денег при отмене или частичной отмене заказа

Не описан механизм возврата средств:

  • Клиент отменил заказ после оплаты → полный возврат Cloud
  • Технолог отменил одну из услуг → частичный возврат
  • Заказ отменён после списания бонусов → возврат в Maxma
  • Промокод использован, но заказ отменён → восстановление промокода?
Что делать: определить политику возвратов — кто инициирует, через какие системы (Cloud refund API, Maxma rollback, АГБИС корректировка). Критично для продакшена.
Среднее · Надёжность интеграций

Отказы внешних систем: АГБИС, Телетрек, Cloud, Life, Maxma

Что делать если:

  • АГБИС недоступен → Этап 1: создание упадёт. Этапы 3 и 5 встанут. Фабричный флоу блокирован.
  • Телетрек недоступен → Этап 1: нет зоны и слотов. Этапы 2 и 6: курьер не назначится.
  • Cloud недоступен → Этап 4 A/B: нет онлайн-оплаты. Этап 6 C-СБП: нет QR.
  • Life недоступен → Этап 6 C-КАРТА: терминал не примет, но можно перейти на СБП.
  • Maxma недоступна → Этап 1: нет скидок в корзине. Этап 4: нет применения лояльности.
Что делать: retry-политика, таймауты, fallback. Критичные операции (создание заказа, оплата) — транзакционная семантика или идемпотентность. Менее критичные — деградация UX с предупреждением.
Среднее · Идемпотентность

Гонки и дубликаты колбэков Cloud / Life

На Этапе 6 могут прилететь дубликаты или оба колбэка (клиент начал СБП, передумал, оплатил картой). Без идемпотентности заказ может быть «оплачен дважды» или зависнуть.

Что делать: в Back — проверка «заказ уже Оплачено» перед обработкой колбэка. В матчинге Life — дедупликация по сумме / времени / курьеру. Cloud-колбэки — по payment_id.
Высокое · Блокирующий вопрос

Неопределённый механизм триггера АГБИС для Этапа 6В (самовывоз)

На Этапе 6В событие «вещи прибыли в ПП, готовы к выдаче» приходит из АГБИС. Но конкретный механизм на сегодня не определён — это может быть: отдельный статус заказа, статус услуги, специальный флаг, отдельное событие в API, либо что-то другое. Без уточнения этот этап нельзя имплементировать.

Аналогично — факт физической выдачи вещей клиенту в ПП тоже приходит из АГБИС, с тем же вопросом.

Что делать: зафиксировать конкретный механизм с командой АГБИС (статус / флаг / webhook / polling). До этого — Этап 6В остаётся на бумаге.
Низкое · UX

Зависимость UX от задержек АГБИС на Этапе 3

Клиент может долго ждать появления фото и комментариев технолога, потом ещё раз — появления кнопки «Подтвердить». До 1.5 часов ожидания от реальных действий технолога до возможности клиента отреагировать.

Что делать: прозрачный индикатор «обновление в течение часа» в UI. Push-уведомление когда данные долетели.

06 · Что не учтено

Сценарии и расширения, осознанно не описанные в текущей версии.

Unhappy paths

Статусы BACK-12…BACK-16, AG-6, AG-I-6, TELE-6, MAX-3 не задействованы. Все эти статусы есть в справочниках, но текущий флоу их не использует. Нужен отдельный документ «Unhappy paths».
  • Клиент не согласен с ценой → отмена на Этапе 3 → возврат вещей
  • Услуга отменена технологом на Этапе 5 → пересчёт → возврат разницы
  • Заказ полностью отменён → возврат всех денег + откат бонусов и промокода
  • Курьер не смог забрать / доставить → перенос, отмена, возврат
  • Платёж не прошёл → повторная попытка / отмена
  • Клиент отказывается принимать вещи на доставке → возврат на фабрику
  • Претензии по качеству после получения → отдельный процесс

Возврат денег

Механизм возврата не описан. Cloud refund, Maxma rollback, коррекция в АГБИС — нужна отдельная документация.

Оформление в пункте приёма

Покрыт как Этап 1Б. Клиент пришёл с вещами в ПП, технолог оформил заказ в АГБИС. Дальше заказ проходит по стандартным этапам с упрощениями: Этап 2 (сбор вещей) пропускается, Этап 3 (приёмка) пропускается, Этап 4 (оплата) — частично в зависимости от выбора клиента.

Получение заказа в пункте приёма

Самовывоз реализован как Этап 6В. Использует «спящие» статусы BACK-10 "Ожидает самовывоз" и APP-9 "Готов к выдаче". Триггеры (готовность + факт выдачи) приходят из АГБИС. Конкретный механизм триггера готовности в АГБИС — пока не определён, требует уточнения с командой АГБИС.

Быстрый заказ

Признак «Быстрый заказ» есть в Back, но не влияет на флоу. Логика приоритизации не описана.

Мониторинг, логирование, аудит

Нефункциональные требования не описаны. Как логировать переходы, какие метрики, как обеспечить аудит — отдельная документация.

07 · Терминология

Соглашения по именованию, которых мы придерживаемся во всей документации.

ТерминЗначение
ЗаказЕдиница работы с клиентом. Состоит из одной или нескольких услуг. Имеет общий статус в каждой системе (APP-*, BACK-*, AG-*, MAX-*).
УслугаКонкретная работа над конкретной вещью (чистка пальто, глажка рубашки). Свой статус в АГБИС (AG-I-*). Клиент видит услуги как детализацию заказа.

ПризнакБулевский или категориальный флаг на заказе, отдельный от статуса. Примеры: признак оплаты, признак подтверждения (фабрика / клиент), признак быстрого заказа.
ВиджетUI-элемент в приложении, временный. W-1, W-2. Имеет жизненный цикл: включается на событии, выключается на другом.
Уведомление (Push)Одноразовое push-сообщение. N-1.
Адрес забораОбязательный адрес, откуда курьер забирает вещи. Указывается на Этапе 1.

СлотВременное окно для курьерской операции. Рассчитывается по зоне адреса (Телетрек).
ЗонаГеографический регион обслуживания. Определяется Телетреком по адресу.
Модуль матчинга LifeКомпонент в Back. Получает от Life уведомления о платежах и сопоставляет с заказами по сумме / времени / курьеру. Нужен потому что Life не знает о заказе.
ЛояльностьТри компонента: скидка (авто), промокод (вводит клиент, применяется авто), бонусы (списываются по решению клиента: всё или 0).
